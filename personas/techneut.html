Referentie-architectuur voor wijkoplossingen

De referentie-architectuur vormt het fundament voor het ontwikkelen en integreren van digitale oplossingen in wijken. Deze architectuur zorgt voor standaardisatie, interoperabiliteit en herbruikbaarheid van componenten.

Architectuuroverzicht

De referentie-architectuur bestaat uit vier hoofdlagen:

Architectuurlagen

  1. Gebruikerslaag - Verschillende interfaces voor eindgebruikers (bewoners, zorgverleners, etc.)
  2. Applicatielaag - Functionele modules en services die specifieke functionaliteiten leveren
  3. Integratie- & Gegevenslaag - Zorgt voor gestandaardiseerde gegevensuitwisseling en dataopslag
  4. Infrastructuurlaag - Hardware, netwerken en basisplatformen

Doorsnijdend zijn de lagen voor beveiliging, privacy en toegangsbeheer die alle vier hoofdlagen omvatten.

Architectuurprincipes

  • Modulair ontwerp - Componenten kunnen onafhankelijk worden ontwikkeld, geïmplementeerd en vervangen
  • Interoperabiliteit - Systemen kunnen effectief gegevens uitwisselen en samenwerken
  • Open standaarden - Gebruik van publiekelijk beschikbare, breed gedragen standaarden
  • Privacy by design - Privacy-overwegingen zijn vanaf het begin geïntegreerd in het ontwerp
  • Toegankelijkheid - Oplossingen zijn toegankelijk voor alle gebruikers, inclusief gebruikers met beperkingen
  • Schaalbaarheid - Architectuur kan meegroeien met het aantal gebruikers en data
  • Beveiliging in elke laag - Beveiligingsmaatregelen zijn geïntegreerd in alle architectuurlagen

Open standaarden voor wijkoplossingen

Het gebruik van open standaarden is essentieel voor het creëren van interoperabele en toekomstbestendige digitale wijkoplossingen. Hieronder volgt een overzicht van de belangrijkste standaarden.

Gegevensuitwisseling in de zorg

Zorg Informatie Bouwstenen (ZIB)

Zorg Informatie Bouwstenen (ZIB's) zijn gestandaardiseerde gegevensbouwstenen voor het vastleggen en uitwisselen van patiëntgegevens.

Voordelen:

  • Eenduidige registratie van zorggegevens
  • Betere interoperabiliteit tussen zorgsystemen
  • Verhoogde herbruikbaarheid van gegevens

Implementatie:

Alle nieuwe wijkoplossingen dienen ZIB's te ondersteunen voor relevante gegevensdomeinen. Voor bestaande systemen is een migratiepad beschikbaar.

ZIB implementatiehandleiding »

FHIR (Fast Healthcare Interoperability Resources)

FHIR is een internationale standaard voor elektronische uitwisseling van zorginformatie.

Voordelen:

  • Moderne webtechnologie (REST, JSON, XML)
  • Flexibel en eenvoudig te implementeren
  • Breed gedragen internationale standaard

Implementatie:

Voor nieuwe wijkoplossingen wordt FHIR R4 aanbevolen als primaire standaard voor API-ontwikkeling. Voorbeeld van een FHIR resource:

{ "resourceType": "Patient", "id": "example", "name": [ { "use": "official", "family": "Jansen", "given": ["Jan"] } ], "gender": "male", "birthDate": "1974-12-25" }

Beveiliging en authenticatie

OAuth 2.0 & OpenID Connect

OAuth 2.0 en OpenID Connect zijn de standaarden voor autorisatie en authenticatie in het wijkplatform.

Implementatie:

Alle digitale wijkoplossingen moeten deze standaarden implementeren voor een uniforme gebruikerservaring en veilige toegang. Dit betekent:

  • Gebruik van JWT (JSON Web Tokens) voor beveiligde informatieoverdracht
  • Ondersteuning voor verschillende authenticatieflows (Authorization Code, Client Credentials)
  • Uitwisseling van gebruikerskenmerken via standaard claims

API-documentatie

De Project Kennisportaal API stelt ontwikkelaars in staat om zorgapplicaties te integreren met het centrale platform. De API is RESTful en gebruikt JSON als data-uitwisselingsformaat.

API Endpoints

Wijkmonitor Data

GET /api/v1/wijkmonitor/data

Opvragen van geanonimiseerde aggregatiedata uit de wijkmonitor.

Parameters:

  • wijk - Wijkcode (optioneel)
  • periode - Jaar en kwartaal (bijv. "2024Q1")
  • categorie - Datacategorie (optioneel)

Voorbeeld request:

GET /api/v1/wijkmonitor/data?wijk=1234&periode=2024Q1 HTTP/1.1 Host: api.projectkennisportaal.nl Authorization: Bearer {token}
Zorgaanbod

GET /api/v1/zorgaanbod

Opvragen van beschikbare zorgaanbod in een wijk.

Parameters:

  • wijk - Wijkcode (verplicht)
  • categorie - Type zorgaanbod (optioneel)
Digitale Hulpmiddelen

GET /api/v1/hulpmiddelen

Opvragen van beschikbare digitale hulpmiddelen.

Parameters:

  • wijk - Wijkcode (optioneel)
  • type - Type hulpmiddel (optioneel)
  • doelgroep - Doelgroep (optioneel)

Authenticatie

De API gebruikt OAuth 2.0 voor authenticatie. Ontwikkelaars moeten eerst een account aanmaken en client credentials verkrijgen.

POST /oauth/token HTTP/1.1 Host: api.projectkennisportaal.nl Content-Type: application/x-www-form-urlencoded grant_type=client_credentials&client_id={client_id}&client_secret={client_secret}

Foutafhandeling

De API gebruikt standaard HTTP statuscodes en retourneert gedetailleerde foutberichten in JSON-formaat:

{ "error": { "code": "RESOURCE_NOT_FOUND", "message": "De gevraagde resource kon niet worden gevonden", "status": 404, "details": { "resource": "wijkmonitor/data", "id": "1234" } } }

Integratiehandleiding

Deze handleiding biedt stapsgewijs instructies voor het integreren van uw applicatie met het Project Kennisportaal ecosysteem.

Integratiemethoden

Er zijn verschillende manieren om te integreren met het platform:

Methode Gebruik Voordelen
REST API Voor directe gegevensuitwisseling Eenvoudig, laagdrempelig, breed ondersteund
WebHooks Voor event-gedreven koppelingen Real-time updates, efficiënt
SMART on FHIR Voor zorgspecifieke toepassingen Standaard in zorgsector, contextrijke integratie
Single Sign-On Voor naadloze gebruikerservaring Verbeterde gebruikerservaring, centraal identiteitsbeheer

Integratie stappenplan

  1. Voorbereiding

    Bepaal welke gegevens en functionaliteiten u wilt integreren en kies de geschikte integratiemethode.

  2. Registratie

    Registreer uw applicatie in het ontwikkelaarsportaal om API-credentials te krijgen.

  3. Ontwikkeling

    Ontwikkel de integratie volgens de technische specificaties en best practices.

  4. Testen

    Test de integratie in de testomgeving en valideer de functionaliteit.

  5. Certificering

    Doorloop het certificeringsproces om te garanderen dat uw integratie voldoet aan alle eisen.

  6. Productie

    Na goedkeuring kan de integratie in productie worden genomen.
